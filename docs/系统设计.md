# KIBot 系统设计

## 总体架构

KIBot 采用前后端分离设计：

- 前端：React/Vite 仪表盘，展示教材列表、图谱、融合队列、Agent 状态等。
- 后端：FastAPI 提供 session、教材、图谱、RAG 等 REST API。
- 存储：每个 session 一个目录，核心状态保存在 `session.json`，上传文件保存在该 session 的 `uploads` 子目录。
- LLM：通过 OpenAI-compatible Chat Completions 协议接入，密钥只从环境变量读取。

## 数据流

1. 用户创建 session。
2. 上传教材文件，后端保存原文件并解析为 `ParsedTextbook`。
3. 解析结果写入 session 的 `textbooks`、`chapters`，chunking 结果写入 `chunks`。
4. 用户选择教材后，图谱与 RAG 只使用 `selected_textbooks` 中的教材。
5. 图谱 API 从 chunk 中抽取概念节点和共现边，写回 `graph_nodes`、`graph_edges`。
6. RAG API 从当前 session 的 chunk 检索证据，可选择调用 LLM 生成答案。
7. 融合决策和报告生成使用同一 session 状态，写入 `integration_decisions`、`report`、压缩统计和 Sankey 数据。

## 会话模型

`KIBotSession` 是后端事实源，包含：

- `session_id`
- `selected_textbooks`
- `textbooks`
- `chapters`
- `chunks`
- `graph_nodes`
- `graph_edges`
- `integration_decisions`
- `messages`
- `memory_summary`
- `token_usage`
- `report`

JSON 存储采用临时文件替换写入，降低写坏 session 文件的风险。session id 使用 UUID 校验，避免路径穿越。

## 教材解析与切片

- 支持文件：PDF、TXT、Markdown、DOCX。
- PDF 解析：PyMuPDF `fitz.open()` 和 `page.get_text()`。
- DOCX 解析：python-docx `Document(path).paragraphs`，再复用章节识别流程。
- 章节识别：匹配 `第X章`、`第X节` 形式标题；未识别时创建“全文”章节。
- chunking：默认 `700` 字符窗口、`80` 字符 overlap。
- chunk 保存教材 id、教材标题、章节、页码范围、内容、字符数，作为图谱和 RAG 的共同基础。

## 知识图谱

图谱构建采用 LLM-first + deterministic fallback：

- `use_ai=true` 且配置 LLM 时，按 chunk 构造 JSON schema prompt，抽取节点和边。
- Prompt 限制只使用教材 chunk，要求节点、关系、证据和 confidence。
- LLM 输出 schema 校验失败时回退规则构图。
- 回退图谱从标题、章节、正文中抽取英文/中文概念 token。
- 同一 chunk 中共同出现的概念形成 `co_occurs` 边。
- 图谱结果写入 session，供前端展示和 RAG 概念匹配复用。

## RAG 检索

RAG 使用当前 session 的选中教材作为边界：

- 无选中教材时返回 `no_selected_textbooks`。
- 检索分数以 BM25 term retrieval 为主，叠加轻量 hashed bag-of-terms embedding 向量检索、章节匹配和图谱概念匹配。hashed vector 使用本地确定性词项向量与 cosine similarity，无需下载外部模型，作为比赛阶段的 lexical embedding fallback。
- 后续可将 hashed vector embedding/vector store 替换为 BGE 等真实 embedding 模型与 FAISS/向量数据库，保持当前 chunk/citation/RAG 接口不变。
- 默认最多返回 5 个 chunk，响应中保留 `bm25_score`、`vector_score`、`chapter_score`、`concept_score` 以便解释检索依据。
- `use_llm=false` 时返回可解释 fallback 摘要。
- `use_llm=true` 且配置 LLM 时，使用检索片段构造上下文；失败时返回 fallback 与安全错误信息。

## 跨教材整合与压缩

整合引擎位于 `backend/services/integration_engine.py`，输入为当前 session 的选中教材、图谱节点和 chunk。它输出：

- `decisions`：每条包含 `decision_id`、`action`、`concept_name`、`sources`、`reason`、`confidence`、`compact_note`。
- `stats`：`original_chars`、`compressed_chars`、`ratio`。
- `sankey`：教材来源到整合概念的流向。

压缩率计算口径：

```text
original_chars = sum(selected_textbook.total_chars)
compressed_chars = sum(len(decision.compact_note) for decision in decisions)
ratio = compressed_chars / original_chars
```

系统目标不是机械截断正文，而是把“整合后的精华讲义体量”定义为所有决策 `compact_note` 的总字符数。引擎会把 `compact_note` 控制在原始字符数的 25%-30% 区间，并在超过 30% 时按预算裁剪，确保赛题要求的 “不超过原始体量 30%” 可计算、可展示、可复核。

## Web 可视化设计

- 中央 ECharts force graph：颜色区分类别/来源，大小反映频次/重要度，形状反映类别。
- 节点点击详情：展示名称、定义、类别、教材来源、章节、页码。
- 搜索高亮：输入概念或章节后，匹配节点保持不透明，其他节点降低透明度。
- Sankey 整合流：展示每本教材中的知识点如何流向整合后的概念。
- RAG 引用展开：每条 citation 可展开原文 chunk，显示相关度分数。

## 评分标准对照

| 维度 | P0/P1 标准 | 当前实现 |
| --- | --- | --- |
| A 文档 | README、需求、系统、Agent、报告 | 五类文档齐全，报告含压缩率和教学完整性 |
| B 解析 | PDF/MD/TXT，进阶 DOCX | PyMuPDF + 文本编码 fallback + python-docx |
| B 图谱 | LLM 抽取 + schema + ≥3 关系 | LLM-first graph prompt + schema 校验 + fallback |
| B 整合 | 语义对齐、决策、≤30% | name/keyword/LLM-ready 双轨说明，merge/keep/remove/split，压缩统计 |
| B RAG | chunk + embedding + 检索 + 引用 | 700/80 chunk + hashed vector embedding + BM25 + 引用展开 |
| C 可视化 | 点击、缩放、拖拽、搜索、多维映射 | ECharts graph + search + node detail + Sankey |
| D Agent | 架构、取舍、RAG pipeline、Prompt | Mermaid、方案对比、Prompt few-shot、防幻觉、路线图 |
| E 工程 | 依赖、错误处理、Docker | 版本锁定、`.env.example`、Dockerfile、docker-compose、魔搭 |
| F 创新 | 额外创新说明 | 成本控制路由、混合检索、session-grounded tools、Sankey、图谱详情 |

## P2 技术报告素材

推荐 P2 主题：`KIBot 的低成本混合 RAG 检索策略`。现有可量化指标：

- 检索策略：term overlap → BM25 → BM25 + hashed vector → BM25 + vector + concept/chapter。
- 评估对象：`tests/test_retriever.py` 中的自动化用例，可扩展为 20-50 个问题。
- 指标：top-1 命中、top-5 召回、平均响应时间、LLM token 输入长度。
- 工程结论：医学教材术语精确匹配强，BM25 保证术语召回；hashed vector 作为零下载 embedding fallback；concept/chapter boost 保证引用章节贴近教学语境。
