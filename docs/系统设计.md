# KIBot 系统设计

## 总体架构

KIBot 采用前后端分离设计：

- 前端：React/Vite 仪表盘，展示教材列表、图谱、融合队列、Agent 状态等。
- 后端：FastAPI 提供 session、教材、图谱、RAG 等 REST API。
- 存储：每个 session 一个目录，核心状态保存在 `session.json`，上传文件保存在该 session 的 `uploads` 子目录。
- LLM：通过 OpenAI-compatible Chat Completions 协议接入，密钥只从环境变量读取。

## 数据流

1. 用户创建 session。
2. 上传教材文件，后端保存原文件并解析为 `ParsedTextbook`。
3. 解析结果写入 session 的 `textbooks`、`chapters`，chunking 结果写入 `chunks`。
4. 用户选择教材后，图谱与 RAG 只使用 `selected_textbooks` 中的教材。
5. 图谱 API 从 chunk 中抽取概念节点和共现边，写回 `graph_nodes`、`graph_edges`。
6. RAG API 从当前 session 的 chunk 检索证据，可选择调用 LLM 生成答案。
7. 融合决策和报告生成使用同一 session 状态，当前字段已预留，接口由并行任务补齐。

## 会话模型

`KIBotSession` 是后端事实源，包含：

- `session_id`
- `selected_textbooks`
- `textbooks`
- `chapters`
- `chunks`
- `graph_nodes`
- `graph_edges`
- `integration_decisions`
- `messages`
- `memory_summary`
- `token_usage`
- `report`

JSON 存储采用临时文件替换写入，降低写坏 session 文件的风险。session id 使用 UUID 校验，避免路径穿越。

## 教材解析与切片

- 支持文件：PDF、TXT、Markdown。
- PDF 解析：PyMuPDF `fitz.open()` 和 `page.get_text()`。
- 章节识别：匹配 `第X章`、`第X节` 形式标题；未识别时创建“全文”章节。
- chunking：默认 `700` 字符窗口、`80` 字符 overlap。
- chunk 保存教材 id、教材标题、章节、页码范围、内容、字符数，作为图谱和 RAG 的共同基础。

## 知识图谱

当前图谱构建是轻量规则版本：

- 从标题、章节、正文中抽取英文/中文概念 token。
- 每本教材最多保留高频概念节点，整体限制可见节点数。
- 同一 chunk 中共同出现的概念形成 `co_occurs` 边。
- 图谱结果写入 session，供前端展示和 RAG 概念匹配复用。

## RAG 检索

RAG 使用当前 session 的选中教材作为边界：

- 无选中教材时返回 `no_selected_textbooks`。
- 检索分数以 BM25 term retrieval 为主，叠加轻量 hashed bag-of-terms embedding 向量检索、章节匹配和图谱概念匹配。hashed vector 使用本地确定性词项向量与 cosine similarity，无需下载外部模型，作为比赛阶段的 lexical embedding fallback。
- 后续可将 hashed vector embedding/vector store 替换为 BGE 等真实 embedding 模型与 FAISS/向量数据库，保持当前 chunk/citation/RAG 接口不变。
- 默认最多返回 5 个 chunk。
- `use_llm=false` 时返回可解释 fallback 摘要。
- `use_llm=true` 且配置 LLM 时，使用检索片段构造上下文；失败时返回 fallback 与安全错误信息。

## 部署边界

当前 docs 分支不修改 Docker、backend、frontend 或 `app.py`。系统设计按当前代码描述已存在能力，并把 integration/report 标记为预期接口，避免在文档中承诺尚未合入本分支的实现。
