# KIBot 需求分析

## 项目目标

KIBot 面向多教材整合场景，帮助教师把多本教材中的章节、概念和证据压缩成可追溯的知识核心。系统强调“可解释、可复核、可部署”：每个回答和整合建议都应回到 session 中的教材、章节、页码和切片证据。

## 用户角色

- 教师/评委：上传教材，查看解析结果、知识图谱、RAG 引用、整合决策和报告草稿。
- 部署者：配置 LLM 服务、运行 FastAPI 与前端、持久化 session 数据。
- 开发者：在单 orchestrator 基础上扩展融合、报告、多 agent 集群等能力。

## 核心需求

1. 教材管理：上传 PDF、TXT、Markdown、DOCX 教材，按 session 隔离存储。
2. 文本解析：使用 PyMuPDF 提取 PDF 文本，使用 python-docx 提取 Word 段落，识别中文章节标题；无法识别章节时以“全文”兜底。
3. 切片索引：按固定窗口和 overlap 生成 chunk，为 RAG 与图谱提供统一证据粒度。
4. 知识图谱：基于选中教材的 chunk 抽取概念节点和共现边，支持前端图谱展示。
5. RAG 问答：只检索当前 session、当前选中教材的 chunk，返回答案、引用和检索片段。
6. 智能体工作区：单 orchestrator 读取结构化 session，通过工具生成状态摘要或调用 LLM。
7. 融合与报告：运行跨教材整合，写入 `integration_decisions`、`stats`、`sankey` 与 `report`，并提供静态样例 `report/整合报告.md`。
8. 部署与安全：通过环境变量配置，不提交真实 API key；session 存储目录可挂载持久化。

## 当前实现状态

已实现：

- FastAPI 应用与 CORS
- React/Vite 仪表盘静态前端
- JSON session storage
- 教材上传、解析、chunking
- 图谱构建与查询 API
- RAG 状态与问答 API
- OpenAI-compatible LLM client
- session-grounded tools 与单 orchestrator
- token usage 数据结构与工具读取

已补齐的评分项：

- 跨教材融合执行 API：`POST /api/integration/run`
- 教师修改融合决策 API：`POST /api/integration/decisions/{decision_id}` 与 `POST /api/chat/message`
- Markdown 整合报告生成 API：`POST /api/report/generate`
- 前端与真实后端的部署链路：Docker + 魔搭创空间 `https://zouweipan00-kibot.ms.show/`

## 非功能需求

- 可追溯：答案和报告尽量携带教材、章节、页码、chunk 引用。
- 可复核：教师修改必须写回结构化 session，而不是只保存在聊天记录中。
- 可扩展：先以单 orchestrator 交付，后续可扩展为教材 agent 集群。
- 可观察：记录 LLM 调用次数、输入 token、输出 token、总 token；无 provider usage 时采用估算。
- 可部署：后端默认监听 `0.0.0.0:7860`，前端可独立开发运行或构建后静态托管。

### P1 加分项

- 混合检索：BM25 + hashed vector embedding + concept/chapter rerank。
- 图谱多维可视化：颜色表示来源/类别，大小表示频次/重要度，形状表示类别，Sankey 展示整合流。
- 搜索与引用展开：图谱搜索高亮，RAG citation 可展开原文 chunk。
- Token 可观测：右侧 Token 面板展示 calls/input/output/total。
- Docker 一键部署：Dockerfile + docker-compose + 魔搭创空间。
- DOCX 解析：额外格式支持。

### P2 挑战项准备

P2 技术报告可围绕“轻量混合 RAG 检索”展开：基线为 term overlap，改进为 BM25 + hashed vector embedding + concept/chapter boost。当前仓库已有 `tests/test_retriever.py` 的自动化对照用例；若赛后补飞书报告，可加入 chunk=300/700/1500 的召回率、平均响应时间与 token 消耗表。

## RAG 分块策略依据

KIBot 选择 `700` 字符窗口、`80` 字符 overlap，原因是医学教材常见段落包含“定义 + 机制 + 临床意义”三段信息。300 字窗口容易截断机制链路；1500 字窗口会把多个概念混在一个 chunk 中，降低引用精度；700/80 在“上下文完整性”和“引用定位”之间折中。

| chunk 设置 | 预期优点 | 预期风险 | KIBot 结论 |
| --- | --- | --- | --- |
| 300/50 | 引用更短 | 机制和因果链容易断裂 | 适合作为精确摘录，不作为默认 |
| 700/80 | 定义、机制、例子大多在同块内 | chunk 数适中 | 默认方案 |
| 1500/120 | 上下文完整 | 召回噪声高、LLM 成本高 | 仅适合报告生成 |
