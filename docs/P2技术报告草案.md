# P2 技术报告草案：低成本混合 RAG 在多教材整合问答中的效果分析

> 状态：决赛冲刺草案。当前文件用于飞书技术报告扩写，实验数据列保留“待测”占位；提交前应替换为真实 benchmark 结果。

## 摘要

KIBot 面向“多本教材压缩为可追溯精华知识库”的任务，核心难点不是单次问答，而是在教材解析、图谱整合、教师复核和 RAG 引用之间保持一致的证据链。本文提出一种低成本混合 RAG 方案：以 700 字 chunk、80 字 overlap 作为默认分块，使用 BM25 捕获精确医学术语，使用本地 hashed vector cosine 作为零依赖向量 fallback，再叠加章节匹配和图谱概念匹配进行 rerank。该方案不依赖外部 embedding 下载和向量数据库，在网络受限或模型 key 不稳定的比赛现场仍能给出 top-5 引用、source chunk 和分项检索分数。配套的 deterministic-vs-LLM 路由把状态类问题从 LLM 调用中剥离，只在解释类问题和自然语言生成中调用模型，从而降低 token 成本并提升演示稳定性。

## 1. 问题分析

赛题要求系统支持多本教材上传、图谱构建、跨教材去重、30% 压缩、RAG 精准问答和教师多轮复核。RAG 子任务的评价重点包括：chunk 元数据完整、检索 top-5、回答必须引用教材名称/章节/页码、找不到答案时不编造。传统纯关键词检索对教材术语重复场景有一定优势，但遇到短 query、章节词与正文词错位、同一概念在多本教材中换一种表述时，排序容易不稳定。纯向量检索理论上更强，但现场部署需要下载 embedding 模型或连接外部 API，可能引入网络、依赖体积、冷启动和成本问题。

因此 KIBot 的目标不是追求最重的检索栈，而是在五小时黑客松约束下获得“足够准、可解释、可复现、可部署”的方案。我们把检索拆成四类信号：BM25 分数表示精确术语相关度；hashed vector 分数表示轻量词项向量相似度；chapter score 表示问题与章节标题的教学语境匹配；concept score 表示问题中出现的图谱概念是否也出现在 chunk 内容或章节中。最终分数为四项相加，并在 API 响应中保留 `bm25_score`、`vector_score`、`chapter_score`、`concept_score`，便于调试和向评委解释。

## 2. 方案设计

### 2.1 分块与元数据

系统在教材解析后将每个章节拆成 chunk。默认参数为 `CHUNK_SIZE=700`、`CHUNK_OVERLAP=80`，符合赛题建议的 500-800 字、50-100 字滑窗。每个 chunk 保留 `textbook_id`、`textbook_title`、`chapter`、`page_start`、`page_end`、`content` 和 `char_count`。选择 700/80 的理由是：医学教材段落通常较长，300 字容易切断定义和例证，1200 字会增加 LLM input tokens；700 字能在引用完整性和成本之间取得平衡。

### 2.2 混合检索

检索流程先过滤当前 session 中被选中的教材，避免未选教材污染答案。随后对 query 和 chunk content 进行 Unicode tokenization，构建 BM25 语料并计算关键词分数。hashed vector 使用 `blake2b` 将 token 投到 256 维 bucket，得到确定性 bag-of-terms 向量，再用 cosine similarity 计算轻量向量分。章节匹配按 query terms 与 chapter terms 的交集计分，图谱概念匹配则从 session graph nodes 中找出 query 命中的概念名，并检查这些概念是否出现在 chunk 内容或章节中。

这个设计的优势是可解释：如果某个 chunk 排名靠前，系统可以展示它是因为 BM25 高、vector 高、章节匹配高，还是因为命中了图谱概念。它的局限也清楚：hashed vector 不是真正语义 embedding，不能充分处理复杂同义词和跨语言对齐。后续可把 `_hashed_embedding_vector()` 替换为 BGE-small-zh 或 sentence-transformers，并用 FAISS/ChromaDB 存储向量，同时保持 API 返回结构不变。

### 2.3 生成与成本路由

RAG 回答阶段有两条路径。若 `use_llm=false` 或 LLM 不可用，系统返回 fallback 摘要，列出检索到的教材片段并保留引用编号。若 `use_llm=true` 且 provider 可用，系统将 top-5 retrieved chunks 注入 prompt，system prompt 明确要求“只使用 retrieved chunks；每个事实声明用 [1] 这样的编号引用；证据不足时说明信息不足”。无论 LLM 是否可用，返回结构都包含 `citations`、`source_chunks` 和 `retrieved_chunks`。

Agent 层另有 deterministic-vs-LLM 路由。对于 status、stats、summary、token、compression、graph 等状态类问题，`KIBotOrchestrator` 直接从 session tool registry 生成 deterministic summary；对于 why、explain、reason 等解释类问题才调用 LLM。这样可以把 token 消耗集中在真正需要自然语言推理的场景，避免对“当前压缩率是多少”“用了多少 token”这类事实问题调用模型。

## 3. 实验设计

### 3.1 Benchmark 构造

建议用 20-50 个教师问题构造 RAG benchmark，覆盖事实性、比较性、推理性和跨教材问题。每个问题标注 expected textbook、expected chapter、expected terms、difficulty 和 expected answer outline。示例：

```json
{
  "question": "炎症反应的核心机制是什么？",
  "type": "事实性",
  "expected_textbook": "病理学",
  "expected_chapter": "炎症",
  "expected_terms": ["血管反应", "防御性反应", "损伤因子"],
  "difficulty": 2
}
```

### 3.2 对比实验

| 实验 | Baseline | 对比方案 | 指标 | 预期结论 |

| 检索策略 | term overlap | BM25、hashed vector、BM25+vector、Hybrid | top-1 命中、top-5 召回、引用准确率、MRR | Hybrid 在章节语境和概念命中上更稳 |

## 4. 讨论

混合 RAG 的工程价值在于它把“准确性”和“可部署性”同时纳入设计。BM25 对医学教材中的固定术语很强，hashed vector 虽然不是真语义模型，但能在无外部依赖时提供稳定的向量相似度补充。chapter/concept boost 则把图谱和教学结构引入检索排序，使系统不是孤立地匹配文本，而是在“当前教材、当前章节、当前概念图谱”的 session 语境中回答问题。

这种方案也有边界。第一，hashed vector 无法处理“白细胞/leukocyte”这类跨语言同义关系，后续应接入 BGE 或 multilingual MiniLM。第二，当前 score 权重是经验值，应通过 benchmark 网格搜索优化。第三，fallback answer 更像证据摘要，不等同于高质量自然语言讲解；正式教学场景仍需要 LLM answer 或教师编辑。第四，压缩质量需要人工评估教学完整性，仅靠压缩比不足以证明课程逻辑不受损。

## 5. 结论

KIBot 的低成本混合 RAG 方案适合黑客松评分语境：它满足 P0 的完整 RAG pipeline 和引用要求，覆盖 P1 的混合检索与 token 可视化，并为 P2 提供可量化的实验主题。若 benchmark 数据证明 Hybrid 在 top-5 召回、引用准确率和成本控制上优于 term overlap 与单一检索信号，则可以说明该方案不是临时拼装，而是一种面向受限部署环境的 AI 工程优化。
